# ARCHITECTURE_GUARD.md

**Unified Electromagnetic Platform – Architecture Guardrail**

> **适用对象**：
> 本文档面向所有参与该代码库的**人类开发者**（核心开发、协作开发、代码评审、维护人员）。
>
> **目的**：
> 明确本平台的**不可破坏架构边界**，防止在长期演进、多人协作、AI 辅助开发过程中发生**结构性退化**。

---

## 1. 本项目是什么，不是什么

### 1.1 本项目是什么

这是一个**统一电磁仿真平台**，同时支持：

* **FDTD**（时域、网格、场更新、GPU 并行）
* **MoM / PEEC / MTL**（频域/时域、积分算子、电路等效）
* **混合仿真**（场–路–线、多物理、多尺度）
* **工业级验证与工程应用**

该平台的核心价值不在于“功能多”，而在于：

> **多种电磁方法在同一工程体系内“长期共存而不相互腐蚀”。**

---

### 1.2 本项目不是什么

* ❌ 不是单一数值方法的实验代码
* ❌ 不是“哪个地方好写就往哪塞逻辑”的脚本集合
* ❌ 不是为了短期性能牺牲长期结构的 demo

---

## 2. 架构第一原则（必须牢记）

### 原则 0：**架构优先于性能，正确性优先于架构**

* 架构错了，性能优化没有意义
* 数学不对，任何优化都是加速错误

---

## 3. 六层架构模型（不可破坏）

所有代码**必须且只能**归属于以下六层之一：

```
L1  Physics Definition        物理定义
L2  Discretization            离散与建模
L3  Operator / Update Eq.     算子 / 更新方程
L4  Numerical Backend         数值后端
L5  Execution Orchestration   执行与耦合编排
L6  IO / Workflow / API       输入输出与流程
```

> **跨层实现 = 架构错误（不是“风格问题”）**

---

## 4. 各层“只允许做什么 / 绝对禁止什么”

### L1. Physics Definition（物理定义层）

**只回答一个问题：物理方程是什么？**

✅ 允许

* Maxwell 方程形式
* 边界条件、端口、激励的物理含义
* MoM / PEEC / MTL / FDTD 的理论模型

🚫 禁止

* 网格索引
* 矩阵结构
* GPU / CPU / 并行逻辑

---

### L2. Discretization（离散层）

**只回答：如何把连续物理空间变成自由度？**

✅ 允许

* Yee 网格、非均匀网格
* 三角网格、Manhattan 网格
* RWG / 高阶基函数

🚫 禁止

* 更新公式
* 积分核
* Solver 调用

---

### L3. Operator / Update Equation（算子层）

这是 **FDTD 与 MoM 真正对齐的关键层**。

✅ 允许

* curl 更新算子
* 积分算子
* 耦合算子
* CPML / source 的数学形式

🚫 禁止

* GPU 优化
* FMM / ACA / H-matrix
* 时间步循环、求解流程

---

### L4. Numerical Backend（数值后端）

**只回答：如何高效计算一个“已定义的算子”？**

✅ 允许

* CPU / GPU kernel
* 稀疏 / 稠密矩阵
* Iterative / Direct solver
* FMM / ACA / GPU BLAS

🚫 禁止

* 引入物理假设
* 改变算子语义
* “悄悄近似而不说明误差”

---

### L5. Execution Orchestration（执行编排层）

这是平台的**中枢神经**。

✅ 允许

* 决定执行顺序
* 管理多物理耦合
* 调度 FDTD pipeline / Hybrid solver
* 管理数据流与生命周期

🚫 禁止

* 写数值 kernel
* 写物理公式
* 偷偷调用 solver 内部细节

---

### L6. IO / Workflow / API

✅ 允许

* CAD / Netlist / Touchstone / HDF5
* Python API
* CLI / GUI
* Workflow 自动化

🚫 禁止

* 改变仿真语义
* 注入隐藏计算逻辑

---

## 5. FDTD 与 MoM 的“思维隔离墙”

| 维度   | FDTD          | MoM / PEEC / MTL |
| ---- | ------------- | ---------------- |
| 数学结构 | 局域、显式         | 全局、隐式            |
| 离散方式 | 网格            | 几何 + 基函数         |
| 时间处理 | 显式步进          | 频域 / 隐式          |
| 后端   | stencil / GPU | matrix / solver  |

> **任何“把一种方法的思维直接套到另一种”的行为，都是高风险的。**

---

## 6. Hybrid / Coupling 的正确理解

* Hybrid **不是“把代码连起来”**
* Hybrid 是：

  * 物理耦合定义（L1–L3）
  * 执行编排（L5）

🚫 **禁止**：
在 Hybrid Solver 中：

* 写积分核
* 写 Yee 更新
* 写 GPU kernel

---

## 7. 典型“看起来合理、实际危险”的行为

❌ 为了省事，在 FDTD kernel 里加电路等效
❌ 为了加速，在 MoM kernel 里写 CUDA thread 索引
❌ 在 source 模块直接用 `dx`（忽略非均匀网格）
❌ 在 Hybrid Solver 中直接调用某个 solver 内部函数
❌ “先这样，之后再重构”（通常永远不会发生）

---

## 8. 代码评审的最低要求（给 Reviewer）

每一个非 trivial 修改，Reviewer 至少要问：

1. 这段代码属于哪一层？
2. 是否跨层？
3. 是否引入了新的隐含假设？
4. 是否会影响 FDTD / MoM 的长期共存？
5. 是否破坏了 Hybrid 的调度权威？

如果回答不清楚 → **拒绝合并**
